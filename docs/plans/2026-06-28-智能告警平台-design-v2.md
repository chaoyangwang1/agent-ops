## 1. 项目背景与设计目标

传统运维告警平台普遍存在**告警风暴、信息孤岛、人工排查耗时、经验依赖严重**等问题。本项目旨在构建一个**以大语言模型驱动的自治代理（AI Agent）为核心**的智能告警平台，实现：

- **降噪与聚合**：从海量告警中提取有效事件。
- **自主分析**：主动调用工具链完成根因推断、影响分析。
- **半自动/全自动处置**：在安全边界内执行自愈动作。
- **对话式交互**：提供 ChatOps 式的自然语言运维入口。

### 2.2 核心设计原则

- **事件驱动**：所有告警、动作、对话均以事件形式在消息总线流转。
- **工具化**：运维能力封装为标准 API，Agent 通过 Function Calling 动态调用。
- **人类可控**：高危操作需审批，全程审计，支持一键降级为纯建议模式。
- **持续学习**：处理结果反哺向量知识库与知识图谱。
- **可观测自身**：平台各组件暴露健康指标，Meta-monitoring 实时监控平台自身状态。

## 3. 数据采集层详细设计

### 3.1 指标采集

**基础设施指标**
通过 Kubelet/cAdvisor 暴露容器 CPU、内存、网络等指标，由 Prometheus Operator 的 ServiceMonitor 自动抓取。

**应用接口指标**
- 使用 Prometheus 客户端库在应用中暴露 `/metrics`，或通过 Istio 自动生成七层流量指标。

**预聚合规则（Recording Rules）**
对于大规模集群，配置 Recording Rules 预先计算常用聚合指标，减少查询延迟：

```yaml
groups:
  - name: precompute
    rules:
      - record: job:http_requests_total:rate5m
        expr: rate(http_requests_total[5m])
      - record: instance:node_cpu:avg5m
        expr: avg by(instance)(rate(node_cpu_seconds_total[5m]))
```

**长期存储与保留策略**
- 各集群 Prometheus 通过 Remote Write 写中心 VictoriaMetrics 集群。
- 外部标签 `cluster` 标识来源。
- 保留策略：原始指标 30 天，聚合指标 12 个月。
- 指标基数限额：每集群 ≤ 1M 活跃时间序列。

### 3.2 日志采集

采用 **双层日志架构**：

```
Fluent Bit → Kafka → Elasticsearch（热数据，7天，全文检索，供 Agent 实时查询）
                   → Loki（冷数据，30天+，标签索引，供长期审计和大范围扫描）
```

- 每个 Node 部署 Fluent Bit DaemonSet，挂载容器日志目录。
- 输出注入 `cluster`、`namespace`、`service` 等标签。
- Agent 的 `search_logs` 工具默认查询 ES（全文索引，P99 < 500ms），
  仅当用户明确查询历史时路由到 Loki。

### 3.3 异构与多行日志处理

**异构日志（级联解析器）**

```conf
[FILTER]
    Name parser
    Match kube.*
    Key_Name log
    Parser json_parser      # 先尝试JSON
    Reserve_Data On

[FILTER]
    Name parser
    Match kube.*
    Key_Name log
    Parser text_parser      # 失败回退正则
    Reserve_Data On
```

- `json_parser`：解析 JSON 日志，直接提取字段。
- `text_parser`：使用正则捕获时间、级别、服务名、消息。
- 始终保留原始 `raw` 字段。

**多行日志（Multiline Parser）**
处理 Java 异常堆栈、Go panic trace、Python Traceback 等：

```conf
[FILTER]
    Name multiline
    Match kube.*
    multiline.parser java,crio,go,python,nodejs
```

- 内置 Java / Go / Python / Node.js 多行 parser 模板。
- 未匹配的用通用 heuristics 兜底（以时间戳或日志级别开头的行视为新事件）。

### 3.4 K8s Events 采集

采用专用 **Kubernetes Event Exporter**，将 K8s Events 推入 Kafka，
与日志共用 Kafka → ES 管道，通过 `event.dataset: "k8s_events"` 标签区分：

```yaml
# kubernetes-event-exporter 配置
receivers:
  - name: events
    k8s:
      namespaces: [".*"]
route:
  routes:
    - match:
        - kind: "Pod|Node|Deployment"
      deliver:
        - kafka:
            topic: "k8s.events"
```

### 3.5 链路追踪采集（Tracing）

采用 **Grafana Tempo**，统一 Grafana 生态（与 Loki/VictoriaMetrics 一致）：

- 应用通过 OpenTelemetry SDK 将 Span 发送到 Tempo Distributor。
- Istio 自动生成服务间调用的 Span。
- 存储后端使用 MinIO 对象存储。
- Agent 的 `query_traces(service, trace_id)` 工具查询 Tempo API，
  获取请求的完整调用链和每跳延迟。

### 3.6 多集群采集方案

采用 **"分散采集 → 中心存储"** 星型架构：

- 每个 K8s 集群独立部署 Prometheus + Fluent Bit + Event Exporter + OTEL Collector。
- 指标通过 remote write，日志/Events 推送至中心存储。
- 所有数据均附带 `cluster` 标签。
- Agent 通过统一 API 查询，可透明跨集群检索。

适配器清单：

## 5. 智能处理层：聚合、降噪与抑制

使用 **Kafka Streams** 实时处理 `raw.alerts`，输出聚合告警到 `aggregated.alerts`，
同时写入 PostgreSQL 和 Elasticsearch，触发 Agent 分析。

### 5.1 机械聚合

基于标签 hash 生成 `aggregation_key`，5 分钟窗口内合并相同告警：
- 同一服务 + 同一指标 + 同一严重级别的告警合并。
- 记录合并数 `merged_count` 和首次/末次触发时间。

### 5.2 语义聚合

在机械聚合后增加语义聚合层，识别不同标签但同一根因的告警：

**第一阶段（Phase 1）：规则引擎**
- 同 Node 上的所有 Pod 告警 → 聚合为 Node 级别告警。
- 同 Deployment 的所有 Pod 告警 → 聚合为 Deployment 级别告警。
- 同 Namespace 的告警 → 聚合为 Namespace 概览告警。

**第二阶段（Phase 2）：拓扑聚合兜底**
- 查询 Neo4j 依赖图，上游已 critical 的下游告警自动标记 `suppressed`。
- 同故障域（同 AZ、同 Node）的告警自动关联。

### 5.3 拓扑抑制与维护窗口过滤

1. **拓扑抑制**：查询 Neo4j 依赖图，若上游已 critical，下游告警自动标记 `suppressed`。
2. **维护窗口过滤**：匹配计划维护期则抑制。
3. **Flapping 处理**：5 分钟内状态变更 ≥ 3 次的告警自动降级为 `warning`。

```
raw.alerts → 机械聚合 → 语义聚合 → 拓扑抑制 → 维护过滤 → Flapping处理 → aggregated.alerts
```

## 6. AI Agent 引擎层

### 6.1 Agent 架构：编排器模式

采用**编排器模式（Orchestrator Pattern）**，与 LangGraph 的 `Send` API 天然契合：

**默认模式：单 Agent**
- 80% 告警由主 Agent 直接处理完成。
- 适用场景：告警明确、依赖链短（< 3 个服务）、有历史匹配案例。

**复杂模式：动态多 Agent**
主 Agent 在以下情况自动 spawn 子 Agent：
- 告警模糊（需多源信息交叉比对）。
- 依赖链长（> 5 个服务，跨多团队）。
- 需并行探索多条诊断路径。
- 需独立核查 Agent 评估高风险操作。

```python
# 编排器决策逻辑
def should_spawn_sub_agents(alert, initial_diagnosis):
    if alert.dependency_depth > 5:
        return True
    if initial_diagnosis.confidence < 0.6:  # 低置信度
        return True
    if len(initial_diagnosis.candidate_causes) > 3:
        return True
    return False  # 默认单 Agent
```

### 6.2 Agent 工作流：假设驱动迭代诊断

使用 **LangGraph** 构建有状态图，核心改进为**假设驱动的迭代诊断循环**：

```
START → alert_received → formulate_hypothesis → tool_executor
         ↑                     ↓                       │
         ├──── 验证循环 ────────┘                       ▼
         │                                    evaluate_evidence
         │                                           │
         │                          ┌─────────────────┤
         │                          ▼                  ▼
         │                  hypothesis_confirmed    new_evidence_found
         │                          │                  │
         │                          ▼                  │
         │                    root_cause_node           │
         │                          │                  │
         └──────── loop (refine hypothesis) ◄───────────┘
                                │
                          decision_node
                           /         \
                     notify          execute
                                      │
                               human_approval?
                                      │
                               execute_node → END
```

**关键状态字段**：

```python
class AgentState(TypedDict):
    alert: dict
    messages: List[dict]
    hypothesis: str              # 当前假设
    hypothesis_history: List     # 历史假设及验证结果
    evidence: dict               # 收集的证据
    diagnosis: str               # 最终诊断
    action_plan: dict
    active_intent: str
    pending_confirmations: List
    protected_data: List[dict]   # 不可压缩的关键数据
```

**超时控制（双维度）**：
1. 最大步数：15 步（工具调用次数）。
2. 最大时长：5 分钟。

达到任一限制后，Agent 输出"阶段性结论"——已排除的假设、剩余疑点、后续建议，
带 `truncated: true` 标记。

### 6.3 工具生态与智能采样

**工具定义（OpenAI Function Calling Schema）**：

```python
tools = [

     "parameters": {"service": "string", "trace_id": "string?"}},
    {"name": "get_topology", "description": "获取服务拓扑依赖（查 Neo4j）",

    {"name": "analyze_blast_radius", "description": "评估故障爆炸半径（图算法）",
     "parameters": {"service": "string", "hops": "int"}},
    {"name": "find_common_dependency", "description": "找两个服务的共同依赖",
     "parameters": {"service_a": "string", "service_b": "string"}},
    {"name": "search_similar_incidents", "description": "向量检索历史相似故障",
     "parameters": {"description": "string", "top_k": "int"}},
    {"name": "get_recent_changes", "description": "查询近期变更记录",
     "parameters": {"service": "string", "hours": "int"}},
    {"name": "restart_pod", "description": "重启 Pod（需 OPA + 人工确认）",
     "parameters": {"namespace": "string", "pod": "string"}},

```

**智能日志采样（Drain 模式聚类）**

替代固定数量截断（如"最多 20 条"），在 `search_logs` 工具内嵌入 Drain 算法：

```
原始 2000 条日志
    ↓
Drain 算法按模式聚类
    ↓
┌──────────────┬──────────────┬──────────────┐
│ 模式A: 800条  │ 模式B: 600条 │ 模式C: 3条    │  ← 异常/低频模式
│ "GET /xxx"   │ heartbeat    │ Error trace   │
└──────────────┴──────────────┴──────────────┘
    ↓ 每模式取 3 条  ↓ 每模式取 2 条  ↓ 全部保留
最终返回：8 条精选日志（覆盖所有模式 + 异常全量）
```

采样策略：
- 高频模式（占比 > 50%）：每模式最多 3-5 条，标注重复次数。
- 低频模式（占比 < 5%）：全部保留（往往是异常信号）。
- 含 ERROR/WARN 关键字的：优先级提升。

### 6.4 分层上下文与记忆管理

利用 **LangGraph Checkpointer** 实现分层上下文窗口：

```
┌──────────────────────────────────────────┐
│ Layer 1: 系统 Prompt（永远保留）           │
├──────────────────────────────────────────┤
│ Layer 2: 工具返回的关键数据（不可压缩）     │  ← protected_data，滑动窗口
├──────────────────────────────────────────┤
│ Layer 3: Agent 推理历史（可压缩）           │  ← 超出阈值时自动摘要
├──────────────────────────────────────────┤
│ Layer 4: 当前上下文窗口（最近 N 轮完整保留） │  ← 保证当前推理连贯
└──────────────────────────────────────────┘
```

- **Layer 2 保护**：工具返回时自标记 `critical` 字段（错误码、异常值、堆栈第一行），进入保护层。
- **Layer 3 压缩**：已排除的假设路径、Agent 推理过程定期摘要，保留"已排除 X，因为 Y"的结论。
- **Layer 4**：最近 5 轮对话完整保留。

```python
# LangGraph Checkpointer + 分层管理
from langgraph.checkpoint.postgres import PostgresSaver

checkpointer = PostgresSaver(conn_string)

# AgentState 中维护分层数据
class AgentState(TypedDict):
    messages: List[dict]
    protected_data: List[dict]       # Layer 2: 不可压缩数据
    compressed_memory: List[str]     # Layer 3: 已摘要历史
    # ... 其他字段
```

### 6.5 异常重试与降级策略

**LLM 调用异常：**

| 错误类型 | 策略 | 示例 |
|----------|------|------|
| 瞬时错误 | 指数退避重试 3 次，间隔 2s/4s/8s | 超时、429 限流、5xx |
| 持久错误 | 不重试，记录并通知 | 400 参数错、401 鉴权 |
| 响应格式错误 | 重试 1 次，仍失败则走兜底规则 | JSON 解析失败、tool_call 不符 Schema |

**工具执行异常：**

| 情况 | 策略 |
|------|------|
| 个别工具失败 | Agent 尝试替代工具或不同查询参数 |
| 全部工具不可用 | 降级为"纯通知模式" |
| 工具返回空结果 | 标注"无结果"（非失败），提示换条件 |

**全局降级：**
- 连续 3 次 LLM 调用失败 → 自动降级为"仅通知模式"，推送原始告警 + 元数据。
- 诊断超时 → 输出阶段性结论，标记 `truncated: true`。
- 自动执行前 OPA 策略检查失败 → 降级为人工审批。

**降级模式定义：**

```python
class AgentMode(Enum):
    FULL = "full"               # 完整诊断 + 自动执行
    DIAGNOSE_ONLY = "diagnose"  # 仅诊断，不执行
    NOTIFY_ONLY = "notify"      # 仅通知原始告警
```

### 6.6 根因定位与问题解决流程

**诊断流程示例**：
1. 收到"支付服务 P99 延迟"告警 → 形成假设："可能是下游依赖问题"。
2. 查询支付服务 CPU/内存（正常）→ 假设弱化。
3. 查询拓扑（依赖 Redis、MySQL）→ 细化假设："可能是 Redis 阻塞"。
4. 查询 Redis 指标（连接超时飙升）→ 假设强化。
5. 检索 Redis 日志（采样后返回含 `KEYS *` 的慢查询日志）。
6. 查询近期变更（有新发布包含 KEYS 命令）。
7. → **综合推断**：代码误用 KEYS 阻塞 Redis 导致延迟，置信度 95%。

**解决动作级别**：
- **Level 0**：仅通知，附分析报告。
- **Level 1**：生成建议（如"建议回滚"），提供一键执行按钮，需人工确认。
- **Level 2**：高度确定且低风险（如重启无状态 Pod），OPA 通过后自动执行，监控恢复情况。

### 8.2 对话持久化与断点恢复

**存储方案：**
- 对话状态存 PostgreSQL（结构化查询、审计）。
- LangGraph checkpoint 存 Redis（快速恢复） + PostgreSQL 持久备份。

**恢复流程：**
1. 用户重新打开 ChatOps → 检测 `status=active` 对话。
2. 提示："检测到未完成的排查「支付服务 P99 延迟」，是否继续？"。
3. 用户确认 → 从 Redis 加载 LangGraph checkpoint → Agent 从断点恢复。
4. 如果原告警已标记 `resolved` → Agent 自动切换为"复盘模式"。

### 8.3 指代消歧三层策略

**Layer 1：槽位绑定（自动）**
用户说"它现在恢复了吗？"，Agent 从 `slot_values` 中取 `current_alert.service` 替代。

**Layer 2：近期上下文（自动）**
槽位无匹配时，回溯最近 3 轮对话提到的实体，按最近优先匹配。

**Layer 3：询问澄清（交互）**
前两层无法消歧时，生成具体反问（最多 3 个候选）：
```
"你是想问「支付服务的 Redis 连接池」还是「支付服务的当前 QPS」的抗压能力？"
```

### 8.4 意图漂移检测与 LangGraph 集成

- 每轮用户输入用轻量 LLM 分类意图（`diagnose`、`execute`、`query`、`chitchat`）。
- 若与当前意图不符：
  - **有未确认危险操作** → 强制用户确认是否放弃。
  - **无风险** → 当前意图压栈，切换至新意图，完成后提示恢复。
- LangGraph 中插入 `intent_guard` 节点，根据检测结果路由。

## 9. 自动化执行与安全控制

### 9.1 OPA 三层策略

**Layer 1：资源边界（硬限制，不可绕过）**

```rego
# 禁止对 StatefulSet/核心中间件执行删除/重启
deny[msg] {
    input.action in ["restart_pod", "delete_pod"]
    input.resource.kind == "StatefulSet"
    msg = "禁止操作 StatefulSet 资源"
}

deny[msg] {
    not input.namespace in data.allowed_namespaces
    msg = "命名空间不在白名单中"
}
```

**Layer 2：影响评估（条件放行）**

```rego
deny[msg] {
    input.affected_replicas / input.total_replicas > 0.5
    msg = "影响范围超过 50%，需人工审批"
}

deny[msg] {
    input.service.error_budget_remaining < 0.05
    msg = "Error Budget 已耗尽，禁止自动变更"
}
```

**Layer 3：时间窗口（变更窗口约束）**

```rego
deny[msg] {
    input.risk_level == "high"
    not now_in_change_window()
    msg = "不在变更窗口内，禁止高风险操作"
}
```

### 9.2 回滚与幂等性

**回滚机制：**
- 每个自动执行动作必须携带 `revert_plan` 字段。
- 执行前快照当前状态（Pod 副本数、ConfigMap Diff）。
- 执行后监控异常恶化 → 自动触发回滚。
- 回滚失败 3 次 → 升级人工介入。

**幂等性设计：**
- 所有执行操作携带 `idempotency_key`。
- 重复执行不会产生副作用（如已重启的 Pod 不再重复重启）。
- 执行前后状态差分为零 → 跳过并记录。

### 9.3 审计与爆炸半径控制

- **全审计**：记录谁（Agent/人）、何时、执行内容、结果、是否回滚。
- **爆炸半径控制**：执行前调用 `analyze_blast_radius` 评估依赖影响。
- **审批流程**：高危操作推送至 ChatOps，待人工点击或二次认证。

### 10.1 API 网关与认证鉴权

- 所有外部请求（ChatOps、Web Console、第三方集成）通过 API 网关。

- 鉴权：基于 RBAC，区分管理员、运维工程师、只读用户、服务账号。
- Token 管理：
  - 支持创建/撤销/轮换 Token，撤销后立即生效。
  - Token 使用记录全量审计（谁、何时、调用什么 API）。
  - 长期未使用的 Token 自动过期提醒。
- 限流：按 Token + API 端点做 Token Bucket 限流。

## 13. 附录：关键代码示例

### Agent 假设驱动诊断循环

```python
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
from langgraph.checkpoint.postgres import PostgresSaver
from langchain_core.messages import AIMessage, HumanMessage
from typing import TypedDict, List, Optional
from enum import Enum
import time

class AgentMode(Enum):
    FULL = "full"
    DIAGNOSE_ONLY = "diagnose"
    NOTIFY_ONLY = "notify"

class AgentState(TypedDict):
    alert: dict
    messages: List[dict]
    hypothesis: str
    hypothesis_history: List[dict]
    evidence: dict
    diagnosis: str
    action_plan: dict
    active_intent: str
    pending_confirmations: List[str]
    protected_data: List[dict]
    compressed_memory: List[str]
    mode: str
    step_count: int
    start_time: float

MAX_STEPS = 15
MAX_DURATION = 300  # 5 分钟

def call_model(state, llm):
    """LLM 调用，含重试逻辑"""
    for attempt in range(3):
        try:
            response = llm.invoke(state["messages"])
            return {"messages": [response]}
        except (TimeoutError, RateLimitError) as e:
            if attempt == 2:
                return degrade_to_notify(state, f"LLM 调用 3 次失败: {e}")
            time.sleep(2 ** attempt)
    return {"messages": [AIMessage(content="LLM 调用失败")]}

def call_tool(state):
    """工具调用，含异常处理"""
    last_msg = state["messages"][-1]
    results = []
    for tc in last_msg.tool_calls:
        try:
            result = tool_executor.invoke(tc)
            # 标记关键数据进入保护层
            if is_critical(tc["name"], result):
                state["protected_data"].append({
                    "tool": tc["name"],
                    "data": result,
                    "timestamp": time.time()
                })
            results.append(FunctionMessage(content=str(result), name=tc["name"]))
        except ToolError as e:
            results.append(FunctionMessage(
                content=f"工具执行失败: {e}，请尝试替代方案",
                name=tc["name"]
            ))
    return {"messages": results, "step_count": state["step_count"] + 1}

def evaluate_evidence(state):
    """评估证据是否支持当前假设"""
    evidence_summary = analyze_evidence_against_hypothesis(
        state["evidence"], state["hypothesis"]
    )
    if evidence_summary["confidence"] > 0.85:
        return "hypothesis_confirmed"
    elif evidence_summary["remaining_hypotheses"] == 0:
        return "exhausted"
    else:
        return "new_evidence_found"

def check_timeout(state):
    """双维度超时检查"""
    if state["step_count"] >= MAX_STEPS:
        return "timeout"
    if time.time() - state["start_time"] > MAX_DURATION:
        return "timeout"
    return "continue"

def formulate_truncated_diagnosis(state):
    """超时时输出阶段性结论"""
    return {
        "diagnosis": generate_partial_diagnosis(state),
        "truncated": True,
        "messages": [AIMessage(
            content=f"诊断超时（{state['step_count']}步/{time.time()-state['start_time']:.0f}秒），"
                   f"已排除 {len(state['hypothesis_history'])} 个假设，"
                   f"剩余疑点: {state['hypothesis']}"
        )]
    }

def degrade_to_notify(state, reason):
    """降级为仅通知模式"""
    return {
        "mode": AgentMode.NOTIFY_ONLY.value,
        "messages": [AIMessage(content=f"降级为通知模式: {reason}")]
    }

workflow.add_node("agent", call_model)
workflow.add_node("tool_executor", call_tool)

workflow.add_node("truncated_diagnosis", formulate_truncated_diagnosis)

workflow.set_entry_point("agent")

workflow.add_conditional_edges("agent", should_continue, {
    "tool_executor": "tool_executor",
    "evaluate": "evaluate_evidence",
    END: END

workflow.add_edge("tool_executor", "evaluate_evidence")
workflow.add_conditional_edges("evaluate_evidence", check_timeout, {
    "timeout": "truncated_diagnosis",
    "skip": "agent",
    "hypothesis_confirmed": "root_cause",
    "new_evidence_found": "agent"
})
workflow.add_conditional_edges("root_cause", check_timeout, {
    "timeout": "truncated_diagnosis",
    "continue": "decision"
})

checkpointer = PostgresSaver(conn_string)
graph = workflow.compile(checkpointer=checkpointer)
```

### 智能日志采样（Drain 聚类）

```python
from drain import Drain

class LogSampler:
    def __init__(self):
        self.drain = Drain(depth=4, max_children=100)

    def sample(self, logs: List[str], max_return: int = 30) -> List[dict]:
        clusters = {}
        for log in logs:
            cluster_id = self.drain.match(log)
            clusters.setdefault(cluster_id, []).append(log)

        sampled = []
        total = len(logs)
        for cluster_id, entries in clusters.items():
            ratio = len(entries) / total
            if ratio > 0.5:
                n = min(3, len(entries))  # 高频模式取 3 条
            elif ratio < 0.05:
                n = len(entries)          # 低频全部保留
            else:
                n = min(2, len(entries))  # 中等频率取 2 条
            sampled.extend(entries[:n])
            sampled[-1] += f" [本模式共 {len(entries)} 条]"

        # 含 ERROR/WARN 的优先保留
        priority = [l for l in logs if "ERROR" in l or "WARN" in l]
        sampled = list(set(priority + sampled))[:max_return]
        return [{"content": s, "critical": "ERROR" in s} for s in sampled]
```
